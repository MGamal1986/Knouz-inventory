## Add "Actual Profit" KPI card to dashboard
**File(s):** apps/api/src/modules/dashboard/dashboard.routes.ts, apps/web/src/pages/Dashboard.tsx
**What changed:** `/api/dashboard/summary` now computes `totalActualProfit` — sum over every `SaleItem` (joined to its `Product`) of `(unitPrice - originalCost) * (quantity - refundedQuantity)`, net of refunds just like the existing revenue calc. Dashboard shows it as a new "Actual Profit" KPI card (green `trending_up` icon) next to Stock Value.
**Why:** User wanted a card like Stock Value but for actual realized profit on sold items, based on the difference between selling price and original cost.
**Impact:** API response gains one field; no breaking changes. Verified against the live dev DB: the API-computed value matched an independent raw SQL aggregate exactly (595), confirmed end-to-end through the same Vite proxy path the browser uses. Both apps typecheck clean; api container rebuilt.
**Status:** Approved

## Fix: deleting a sold-out product silently failed; switched Product delete to soft-delete
**File(s):** apps/api/prisma/schema.prisma, apps/api/src/modules/products/products.service.ts, apps/api/src/modules/inventory/inventory.routes.ts, apps/api/src/modules/dashboard/dashboard.routes.ts, apps/api/src/modules/sales/sales.service.ts, apps/web/src/pages/Inventory.tsx
**What changed:** Discovered `DELETE /api/products/:id` was hard-deleting via `prisma.product.delete()`, but `SaleItem.productId` has `ON DELETE RESTRICT` in Postgres — so any sold-out product (which by definition has a `SaleItem` referencing it) could never actually be deleted; it failed with a 500 FK-constraint error, and the frontend had no error handling so this failed silently with the item staying put everywhere. Added a nullable `deletedAt` column to `Product` and switched `deleteProduct()` to soft-delete (sets `deletedAt`, keeps the row). Filtered `deletedAt: null` into: product listing/search (`listProducts`, `getProductByCode`), the Inventory page's product query, and the dashboard's `/summary` (product/stock counts, category stats) and `/sold-out` queries. Left the profit calc's `SaleItem` query unfiltered so historical profit from a later-deleted product still counts. Added `deletedAt` guards to `updateProduct`/`restockProduct` (404 if already deleted) and to sale creation (can't sell a deleted product). `Inventory.tsx`'s `onDelete` now wraps the API call in try/catch and alerts the error instead of failing silently.
**Why:** User asked to verify that deleting a sold-out item from Inventory also removes it from the dashboard's restock/Sold Out section. Testing (created a throwaway product + sale + sale-item mirroring a real sold-out item, then hit the real delete endpoint) proved it currently didn't work at all — confirmed via docker inspection that the FK is `ON DELETE RESTRICT`. User's follow-up requirement: deleting should also never let a new product reuse the deleted item's code — already guaranteed independently by `generateNextProductCode`'s per-category `lastSeq` counter (monotonic, never derived from existing rows), and reinforced further since soft-delete keeps the unique `productCode` occupied forever.
**Impact:** Schema change: added nullable `Product.deletedAt`, pushed to the dev DB via `prisma db push` (project has no migrations directory — schema changes are pushed directly). No data loss: sale/invoice/profit history for deleted products stays fully intact since `Sale`/`SaleItem`/`Refund` rows are untouched. Verified live end-to-end against the running containers: rebuilt the api image, created a real sold-out test product with a real sale against it, confirmed delete now returns 200 (previously 500), confirmed it disappears from Inventory search, the dashboard's Sold Out list, and dashboard product/stock counts, confirmed `totalActualProfit` was unchanged before/after (730 → 730, since the `SaleItem` row survives), confirmed a double-delete correctly 404s instead of crashing, and confirmed the deleted product can no longer be found by `/api/products/search` or sold. All test fixtures (product, sale, sale item) and the temporarily-reset admin password were cleaned up afterward.
**Status:** Approved
