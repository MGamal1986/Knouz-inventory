## Add per-product discount (percentage or fixed EGP), applied at checkout and on invoices
**File(s):** apps/api/prisma/schema.prisma, apps/api/src/costing/pricing.ts, apps/api/src/modules/products/products.service.ts, apps/api/src/modules/products/products.routes.ts, apps/api/src/modules/sales/sales.service.ts, apps/api/src/modules/sales/sales.routes.ts, apps/api/src/lib/pdf.ts, apps/web/src/components/ProductFormModal.tsx, apps/web/src/pages/Inventory.tsx, apps/web/src/pages/NewSale.tsx, apps/web/src/pages/InvoicePreview.tsx
**What changed:** Added `DiscountType` enum (`NONE`/`PERCENTAGE`/`FIXED`) plus `discountType`/`discountValue` on `Product`, set from the Add/Edit Product form (with a live "price after discount" preview) and shown in the Inventory table. `SaleItem` now snapshots `originalUnitPrice`, `discountType`, `discountValue`, and `discountAmount` at sale time (in addition to the existing discounted `unitPrice`/`lineTotal`), so historical invoices stay accurate even if the product's discount changes later. At checkout (`NewSale.tsx`), a product with a discount auto-applies it (struck-through original price shown), and — per user decision — each line has a checkbox to toggle the discount off for that specific sale only (`applyDiscount` flag sent per line item to `POST /api/sales`). Both the on-screen `InvoicePreview` and the generated PDF (`pdf.ts`) show a Discount column per line plus a Subtotal/Discount/Total breakdown. `calculateDiscountedPrice()` in `pricing.ts` clamps the result to `[0, price]` so a bad/oversized discount can never go negative.
**Why:** User wants to be able to discount any inventory item (either as a % or a flat EGP amount), have it apply automatically during checkout, and have it reflected on the invoice.
**Impact:** Schema migration required (no committed `prisma/migrations` directory exists in this repo — see standalone note below; applied via `prisma db push` against the live dev DB). `SaleItem.unitPrice`/`lineTotal` keep their existing meaning (post-discount, actually-charged values), so nothing downstream that already read those fields breaks. Verified end-to-end against the live dev DB: created throwaway products/client/sale (percentage discount, fixed discount, and one line with the discount toggled off), confirmed computed `unitPrice`/`discountAmount`/`lineTotal`/`totalAmount` were exactly correct, confirmed PDF generation succeeded, then deleted all test rows/PDF/throwaway admin account — real data (57 products) was untouched throughout.
**Status:** Approved

## Add sale refunds (restock + revenue deduction) and net-of-refund dashboard summary
**File(s):** apps/api/prisma/schema.prisma, apps/api/src/modules/sales/sales.service.ts, apps/api/src/modules/sales/sales.routes.ts, apps/api/src/modules/dashboard/dashboard.routes.ts, apps/web/src/components/RefundModal.tsx (new), apps/web/src/pages/Sales.tsx
**What changed:** Added a `Refund` model (linked to `SaleItem`; records quantity, `refundAmount`, optional reason, timestamp) and `SaleItem.refundedQuantity`. New `POST /api/sales/:id/refunds` accepts `{ items: [{ saleItemId, quantity }], reason? }`, refunds any quantity up to what's still refundable per line (partial refunds allowed, per user decision), and per refunded unit: creates a `Refund` row, increments `refundedQuantity`, and decrements the product's `quantitySold` (restocking it) while recalculating its status — the product's `originalCost`/`sellingPrice`/`discountType`/`discountValue` are never touched by a refund, only stock quantity. The amount deducted from revenue is the price actually charged at sale time (`unitPrice`, i.e. post-discount if a discount applied) — per user decision, not the undiscounted list price. `listSales`/`getSaleById` now also return computed `netTotal` (= gross minus refunded) and `totalRefunded` per sale. `Sales.tsx` gained a refund icon per row opening `RefundModal` (shows sold/already-refunded/refundable qty per line, quantity inputs, optional reason), and displays the net total (original struck through) plus a "Refunded" badge when applicable.
**Why:** User wants a way to process returns on any past sale — putting the product back into inventory (at its existing, unmodified cost/price) and removing the actually-collected amount from revenue.
**Impact:** Additive schema change (same `prisma db push` caveat as above). Over-refund attempts are rejected with a 400 listing the exact refundable quantity (verified). Verified end-to-end on the live dev DB: sold 4 units of a 10%-off test product (135/unit), refunded 1, confirmed `quantitySold` dropped from 4→3, product pricing/discount fields unchanged, `netTotal` became 405 (540 − 135), then cleaned up all test rows/PDF/throwaway admin.
**Status:** Approved

## Add filterable "Revenue Explorer" to Dashboard (time / category / client / product)
**File(s):** apps/api/src/modules/dashboard/dashboard.routes.ts, apps/web/src/components/RevenueExplorer.tsx (new), apps/web/src/pages/Dashboard.tsx
**What changed:** New `GET /api/dashboard/revenue?from=&to=&categoryId=&clientId=&productId=` sums matching `SaleItem`s and returns `grossRevenue`, `refundedAmount`, `netRevenue`, `salesCount` — always net of refunds. Added a "Revenue Explorer" card on the Dashboard (Time Range preset: Today/Week/Month/Year/All Time/Custom + Category/Client/Product selects) that calls this endpoint on any filter change. The existing "Revenue This Month" KPI on `/api/dashboard/summary` was also changed to be net of refunds (previously summed raw `Sale.totalAmount`, which ignored refunds entirely), keeping it consistent with the explorer.
**Why:** User wants to be able to slice the dashboard revenue figure by time range, category, client, or product.
**Impact:** `revenueThisMonth` in the summary payload will now read lower than before for any month that has refunds against it (this is the intended fix, not a regression — it was overstating revenue previously). Verified filtering by product, client, and category independently all returned the correct net figures on a live test sale+refund, then cleaned up. Both apps typecheck clean (`tsc --noEmit`).
**Status:** Approved

## Note: no committed Prisma migrations in this repo (discovered, not fixed)
**File(s):** apps/api/prisma/ (missing `migrations/`), apps/api/Dockerfile
**What changed:** Nothing — this is a pre-existing condition, flagged for visibility. The repo has no `prisma/migrations` directory under version control, yet the Docker image's startup command runs `npx prisma migrate deploy`, which is a no-op with no migrations to apply. The live dev DB's `_prisma_migrations` table shows a single `20260709075529_init` migration that was applied at some point but never committed. All schema changes this session (and, per that history, prior sessions) were applied directly via `prisma db push` against the running Postgres container.
**Why:** Surfaced while adding the discount/refund schema changes — wanted to confirm the correct way to apply schema changes in this project before doing so.
**Impact:** `prisma migrate deploy` in the Docker image currently does nothing; if the DB is ever rebuilt from scratch (fresh volume), it will start with **no schema at all** unless someone runs `db push` (or migrations get properly initialized) against it first. Not fixed this session since it's outside the requested scope — worth addressing deliberately (e.g. `prisma migrate dev --name init` against a clean DB, committing the resulting `migrations/` folder) before this project is deployed anywhere beyond the current dev environment.
**Status:** Proposed
